01state はスナップショット

「スナップショット」とは写真のことです。写真は撮った瞬間の状態を切り取ったものであり、その後に被写体が動いても写真は変わりません。React の state も同じです。

レンダリングが始まった瞬間、React はそのときの state の値を「写真に撮って」固定します。 イベントハンドラーの中でどれだけ state を読んでも、その「写真」の値が返ってきます。

レンダリングとは何か
React がコンポーネント関数（function Counter() { ... }）を呼び出して JSX を作る作業のことです。ボタンをクリックして setState が呼ばれると、React は次のレンダリングをスケジュールし、次の呼び出しで新しい JSX が作られます。

コードで確認する

function Counter() {
  const [count, setCount] = useState(0);

  function handleClick() {
    // ここで count は「このレンダリング時点」の値（例: 0）で固定されている
    console.log(count); // → 0
    setCount(count + 1);
    console.log(count); // → 0（まだ変わっていない！）
    setCount(count + 1);
    console.log(count); // → 0（やっぱり 0）
  }

  return (
    <button onClick={handleClick}>
      count = {count}
    </button>
  );
}

上のコードで「増やす」ボタンを押すと、count は 2 ではなく 1 しか増えません。なぜなら:

  • このレンダリングでは count は 0 として固定されている
  • setCount(count + 1) は setCount(0 + 1) つまり setCount(1) と同じ
  • 何度呼んでも「次のレンダリングで count を 1 にしてください」という命令が並ぶだけ
setCount は「今すぐ count を変える」関数ではない
setCount は「次のレンダリング時に使う値をリクエストする」関数です。呼んだ直後に count を読んでも、値は変わっていません。

02同じ setState を何度呼んでも1回しか増えない理由

前のセクションの例を図で整理します。ボタンをクリックしたとき React の中で何が起きているかを追いかけてみましょう。

初期状態: count = 0

ボタンをクリック → handleClick が実行される：

実行される行React が受け取る命令count の値（このレンダリング）
setCount(count + 1)「次は 1 にして」0（変わらない）
setCount(count + 1)「次は 1 にして」（同じ！）0（変わらない）
setCount(count + 1)「次は 1 にして」（同じ！）0（変わらない）

3回とも count + 1 は 0 + 1 = 1 なので、「次は 1 にして」という命令が3回届くだけです。結果: 次のレンダリングで count は 1。

なぜこの仕組みにしているのか
1回のクリックの中で state が途中で変わると、「途中の状態」でコンポーネントが混乱します。React は1回のイベントが終わるまで state を固定することで、処理を安定させています。これを「state の一貫性」と呼びます。

03バッチ処理（React がまとめて処理する）

バッチ（batch） とは「まとめる」という意味です。React は1つのイベントハンドラーの中で複数の setState が呼ばれても、すべての setState が終わってから、まとめて1回だけ再レンダリングします。

function handleClick() {
  setCount(count + 1);  // ① リクエストをためる
  setIsOn(!isOn);          // ② リクエストをためる
  setText("hello");       // ③ リクエストをためる
  // ④ ここで全部まとめて再レンダリング（1回だけ）
}

これを バッチ処理 といいます。バッチ処理のおかげで、「途中の中途半端な状態」が画面に表示されることがなく、パフォーマンスも良くなります。

React 18 からは「自動バッチ」が強化された
React 18 より前は、イベントハンドラーの中だけバッチ処理されていました。React 18 からは setTimeout や fetch の中でも自動的にバッチ処理されるようになりました。今は「React は基本的にまとめて処理する」と覚えておけば大丈夫です。

04関数型更新（updater function）

では「本当に count を3増やしたい」とき、どうすればよいでしょうか。答えは 関数型更新 です。

setState に値ではなく 関数 を渡すと、React は「前の state の値（最新のもの）」をその関数の引数に渡してくれます。

❌ 値を渡す（スナップショットを使う）
// count が 0 の場合
setCount(count + 1); // → 1 にして
setCount(count + 1); // → 1 にして（同じ）
setCount(count + 1); // → 1 にして（同じ）
// 結果: count = 1
✅ 関数を渡す（最新値を使う）
// c には「前の値」が入ってくる
setCount(c => c + 1); // c=0 → 1 にして
setCount(c => c + 1); // c=1 → 2 にして
setCount(c => c + 1); // c=2 → 3 にして
// 結果: count = 3

c => c + 1 の c はアロー関数の引数で、React が「前の値」を入れてくれます。変数名は何でも構いません（prev、n、prevCount など）。慣習として短い名前がよく使われます。

更新キューの仕組み

関数型更新を使うと、React は「更新キュー」に関数を積みます。キューとは「先入れ先出し」の順番待ちのことです。

c => c+1
→
c => c+1
→
c => c+1
→
count = 3

React はキューの関数を順番に実行し、前の関数の結果を次の関数の c に渡します。これにより3回分が正しく積み上がります。

05通常の更新 vs 関数型更新 — 使い分け

どちらを使えばよいのか、判断基準をまとめます。

状況使う書き方理由
単純に値をセットする（入力値、トグルなど） setValue(newValue) 前の値に依存しないので通常の書き方でよい
前の値を基に計算する（カウンター、配列への追加など） setValue(v => v + 1) 前の値が確実に最新のものになる
1つのイベントで同じ state を複数回更新する setValue(v => v + 1) スナップショット問題を避けるため
迷ったら関数型更新を使えばほぼ安全です
「前の値に依存していないのに関数型更新を使っても問題はない」という判断で、関数型更新を基本にするスタイルも一般的です。Day 08 で書いたカウンターコードも setCount(c => c + 1) の方が厳密には正しいです。

トグルでの例

✅ どちらでも動く（値渡し）
// isOn のスナップショットを使う
setIsOn(!isOn)
✅ より安全（関数型更新）
// 必ず前の値を反転させる
setIsOn(v => !v)

06まとめ

概念ひとことで
スナップショットレンダリング時点の state の値は固定。イベントハンドラー内で変わらない
バッチ処理1つのイベント内の setState は全部まとめてから1回だけ再レンダリングされる
関数型更新setState(v => v + 1) で「前の値」を確実に受け取って計算できる
更新キュー関数型更新を積み重ねると、React が順番に実行してくれる
今日の核心は「setStateは次のレンダリングへのリクエスト」という1文です。
これが腑に落ちると、React のバグの多くが「あ、スナップショットの問題だ」と瞬時にわかるようになります。演習で手を動かして確認しましょう！
