1イベントハンドラーの書き方

「ユーザーが何かした瞬間に実行したい処理」を イベントハンドラー と呼びます。 「クリックされた」「テキストを入力した」「フォームを送信した」などのタイミングで動く関数です。

onClick で「クリックを聞く」

React では、HTML タグに onClick（キャメルケース！）という props を渡すと、クリック時に指定した関数を呼んでくれます。

function MyButton() {
  function handleClick() {
    console.log('クリックされた！');
  }

  return (
    <button onClick={handleClick}>
      クリック
    </button>
  );
}
命名の慣習: イベントハンドラーの関数名は handle○○ と書くのが React の一般的な慣習です。 handleClick、handleChange、handleSubmit のように。

⚠️ 超重要：「関数を渡す」と「関数を呼ぶ」の違い

これは React 入門で一番ハマりやすいポイントです。関数は「渡す」のであって「呼ぶ」のではないことを覚えてください。

❌ 間違い — クリックしなくても即実行される
<button onClick={handleClick()}>
✅ 正しい — クリック時だけ実行される
<button onClick={handleClick}>

なぜそうなるかというと——

  • handleClick → 「この関数のことを React に教える」（関数の参照を渡す）
  • handleClick() → 「今すぐ関数を実行して、その戻り値を渡す」（JSX が評価された瞬間に実行される）

つまり onClick={handleClick()} と書いてしまうと、ページを表示した瞬間に全部実行されてしまいます。

アロー関数でインラインに書く

短い処理なら、アロー関数を直接書くこともできます。

<button onClick={() => console.log('押した！')}>
  クリック
</button>

アロー関数の場合は () => ... という形なので「関数を渡している」ことになります（ () をつけても即実行にはなりません）。

なぜインラインアロー関数は OK なのか？
() => handleClick() と書いた場合、これは「クリックされたときに handleClick() を呼ぶ、という新しい関数」です。 即実行はされません。複数の処理をまとめたいときや、引数を渡したいときに便利です。

onChange で「テキスト入力を聞く」

テキストボックスに入力されるたびに動かしたいときは onChange を使います。 イベントハンドラーは イベントオブジェクト（慣習として e や event という名前）を自動で受け取れます。 e.target.value で、今入力されているテキストを取得できます。

function MyInput() {
  function handleChange(e) {
    // e.target は入力欄の DOM 要素
    // e.target.value は今入力されている文字列
    console.log(e.target.value);
  }

  return <input onChange={handleChange} />;
}
▶ ライブデモ — イベントオブジェクトを確認する
主なイベント属性一覧: onClick（クリック）、 onChange（入力値の変化）、 onSubmit（フォーム送信）、 onMouseEnter / onMouseLeave（ホバー）。 HTML の onclick と違い、React は キャメルケース です。

2useState の基本 ― フックとは

フック（Hook）とは

React には フック（Hook） と呼ばれる特別な関数群があります。use で始まる名前が特徴です。 フックは「React の機能をコンポーネントの中で使うためのスイッチ」のようなものです。

今日学ぶ useState もフックの1つで、コンポーネントに「状態（state）を覚えさせる」機能を追加します。

フックのルール（重要）:
  • フックは必ず コンポーネント関数の最上位 で呼ぶ
  • if 文、for 文、入れ子の関数の中では呼べない
  • フックを呼べるのは React のコンポーネント（または後で学ぶカスタムフック）の中だけ

「普通の変数」ではダメな理由

まず「なぜ普通の変数ではいけないのか」を理解しましょう。

// ❌ これは動かない
function Counter() {
  let count = 0;  // ← ただの変数

  return (
    <div>
      <p>カウント: {count}</p>
      <button onClick={() => { count = count + 1; }}>増やす</button>
    </div>
  );
}

変数を書き換えても、React は「何かが変わった！画面を更新しよう」とは気づいてくれません。 React が画面を更新（再レンダリング）するには、React が知っている仕組みを使う必要があります。 それが useState です。

useState の構文

const [count, setCount] = useState(0);
//     ↑今の値  ↑更新関数     ↑初期値（最初の1回だけ使われる）

3点を押さえましょう：

  • useState(0) ：初期値 を渡す。数値・文字列・真偽値・配列・オブジェクトなんでも OK。
  • count（1つ目） ：現在の state の値。これを JSX の中で表示に使う。
  • setCount（2つ目） ：state を更新するための関数。これを呼ぶと画面が更新される。
分割代入の復習（Day02）: useState は内部的に [値, 関数] という配列を返しています。 const [count, setCount] = ... は配列の分割代入で、1番目を count、2番目を setCount という名前に取り出しているだけです。

カウンターの完全なコード

function Counter() {
  const [count, setCount] = useState(0);  // ← 初期値 0

  return (
    <div>
      <p>カウント: {count}</p>
      <button onClick={() => setCount(count + 1)}>
        増やす
      </button>
    </div>
  );
}

3state が変わると何が起きるか（再レンダリング）

setCount を呼んだとき、内部で何が起きているかを理解しておきましょう。

  1. ボタンをクリック → onClick ハンドラーが実行される
  2. setCount(count + 1) が呼ばれる
  3. React が「count の state が変わった！」と検知する
  4. React が Counter 関数を もう一度実行（再レンダリング） する
  5. useState(0) は「初回だけ 0 を使う」ので、2回目以降は新しい値を返す
  6. 新しい count の値で JSX が作られ、画面が更新される
再レンダリング = コンポーネント関数の再実行:
React の再レンダリングとは、コンポーネント関数をもう一度最初から実行することです。 const [count, setCount] = useState(0) の行も毎回通りますが、 React は「このコンポーネントの state はすでにある」と記憶していて、初期値 0 ではなく更新後の値を返してくれます。

前の値をもとに更新する

今回のカウンターは setCount(count + 1) と書きました。 これは「今の count に 1 を足した値にする」という意味です。 多くの場合これで十分ですが、「前の値を確実に使いたい」ときは 更新関数（関数形式） を使います。

// 普通の書き方（多くのケースでこれで OK）
setCount(count + 1);

// 関数形式（直前の state 値を確実に受け取る）
setCount(prev => prev + 1);

関数形式は Day09「state のスナップショット」で詳しく扱います。今日はまず基本形で慣れましょう。

4テキスト入力を state で管理する

ユーザーの入力をリアルタイムで画面に反映させるには、入力値を state で管理するパターンを使います。 これを React では「制御されたコンポーネント（Controlled Component）」と呼びます。

function NameInput() {
  const [name, setName] = useState("");  // 初期値は空文字

  return (
    <div>
      <input
        value={name}                          // ① state の値を input に反映
        onChange={(e) => setName(e.target.value)} // ② 入力されるたびに state を更新
      />
      <p>こんにちは、{name} さん！</p>
    </div>
  );
}

仕組みを整理すると：

  1. ユーザーがキーを押す → onChange が発火 → setName(e.target.value) で state を更新
  2. React がコンポーネントを再レンダリング
  3. value={name} が新しい state の値で更新される → 画面に反映
value props をつけた input は「React が管理する」:
value={name} を渡すと、その input の中身は常に name state と同期します。 もし onChange なしで value だけ渡すと、入力しても文字が変わらない input になります（read-only 的な挙動）。

5複数の state を使う

1つのコンポーネントに useState をいくつでも書けます。 管理したい状態の数だけ書けば OK です。

function ProfileForm() {
  const [name, setName]   = useState("");
  const [age,  setAge]    = useState(0);
  const [isAdult, setIsAdult] = useState(false);

  return (
    <div>
      <input
        placeholder="名前"
        value={name}
        onChange={(e) => setName(e.target.value)}
      />
      <input
        type="number"
        value={age}
        onChange={(e) => setAge(Number(e.target.value))}
      />
      <p>{name} さん、{age} 歳{isAdult ? "（成人）" : ""}</p>
    </div>
  );
}
何を state にすればいい？
「ユーザーの操作によって変わる値」「時間とともに変わる値」は state にします。 変わらない値（固定のラベル文字列、設定値など）は state にせず、単なる変数や props にします。

配列を state にする

アイテムのリストを state で管理することもできます。 配列を更新するときは 元の配列を直接変更してはいけません。 スプレッド構文（Day02）を使って新しい配列を作ります。

function List() {
  const [items, setItems] = useState([]);

  function addItem() {
    // ❌ items.push("新しい項目") ← 直接変更はダメ
    // ✅ 新しい配列を作って渡す
    setItems([...items, "新しい項目"]);
  }

  return (
    <div>
      <button onClick={addItem}>追加</button>
      <ul>
        {items.map((item, i) => <li key={i}>{item}</li>)}
      </ul>
    </div>
  );
}
なぜ直接変更してはいけないのか？
items.push(...) のように元の配列を変更してしまうと、React は「配列が変わった」と検知できません。 React は「前回の値と今回の値を比較して変化を検知」するため、新しい配列を作って渡す必要があります。 詳しくは Day10 で扱います。

今日のキーワード整理

用語意味例
イベントハンドラー ユーザーのアクションに応じて実行される関数 handleClick、handleChange
onClick / onChange React の prop。イベントハンドラーを登録する <button onClick={fn}>

よくある間違い

間違い正しい書き方なぜ？
onClick={fn()} onClick={fn} fn() は即実行。関数の参照を渡すべき
count = count + 1 setCount(count + 1) 直接変更では React が変化を検知できない
items.push(x); setItems(items) setItems([...items, x]) 元の配列を変更してから渡しても変化を検知できない
if (x) { useState(...) } 必ずコンポーネントの最上位で呼ぶ フックのルール：条件分岐の中では呼べない
value={name} だけで onChange なし 必ず onChange と一緒に使う onChange がないと入力を受け付けない read-only になる
