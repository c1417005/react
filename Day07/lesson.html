00なぜ「条件分岐」と「リスト」が必要か

Day06までは、コンポーネントは渡された Props をそのまま表示するだけでした。 しかし実際のアプリでは「ログインしている人にだけメニューを出す」「投稿が0件のときは『まだありません』と表示する」「ユーザー一覧を1人ずつカードにして並べる」—— こうした状況に応じて表示そのものを変える場面が大量に出てきます。

JSXは見た目こそHTMLに似ていますが、中身はただのJavaScriptの式です。 だから「JSX専用の特別な条件分岐構文」は存在しません。普段のJavaScriptで使う if・三項演算子・&&・map() が、そのままReactでも主役になります。

覚えておくこと: Reactには「もし〜なら表示」という専用構文はありません。 「JSXを作るかどうか」「どのJSXを作るか」を、ふつうのJavaScriptの条件分岐で決める——これが条件付きレンダリングの正体です。

01if 文で出し分ける

一番素朴な方法は、コンポーネントの中で関数の早期 return（途中で結果を返す）を使うことです。 Day01で学んだ「関数は return した時点で処理を終える」という性質をそのまま使います。

function Greeting({ isLoggedIn }) {
  if (isLoggedIn) {
    return <h1>ようこそ、戻ってきましたね！</h1>;
  }
  return <h1>サインインしてください。</h1>;
}

isLoggedIn という props（真偽値）の値によって、そもそも別のJSXをreturnするというシンプルな発想です。 「画面の一部を変える」のではなく「コンポーネントが返す結果を丸ごと変える」と考えると分かりやすいでしょう。

1行で書きたいときは早期returnが便利。 「条件に当てはまらなければ何も表示しない」場合は、 if (!data) return null; のように先頭でさっさと抜ける書き方がよく使われます（null の意味はセクション4で説明します）。

02三項演算子 condition ? A : B

if文は「文（statement）」なので、JSXの中（{ } の中）には直接書けません。 JSXの中で条件分岐をしたいときは、式（expression）である三項演算子を使います。

function Item({ isPacked, name }) {
  return (
    <li>
      {isPacked ? name + " ✅" : name}
    </li>
  );
}

条件 ? 真のときの値 : 偽のときの値 という形です。 日本語にすると「isPacked が true なら『name + ✅』、そうでなければ『name』」。 if文と違ってこれ自体が1つの値（式）なので、{ } の中にそのまま埋め込めます。

JSXそのものを切り替えることもできる

function Item({ isPacked, name }) {
  return (
    <li>
      {isPacked ? (
        <del>{name}</del>
      ) : (
        name
      )}
    </li>
  );
}
if文 と 三項演算子、どちらを使う？ 「コンポーネント全体を丸ごと出し分ける」ときは if文（セクション1）、 「JSXの一部分だけを出し分ける」ときは三項演算子、と使い分けるのが定番です。

03論理 AND && で「あるときだけ」表示

「条件がtrueのときだけ表示し、falseのときは何も表示しない」——この場合、三項演算子で : null と書くより、 もっと短く書ける方法があります。論理AND演算子 && です。

function Item({ isPacked, name }) {
  return (
    <li>
      {name} {isPacked && "✅"}
    </li>
  );
}

JavaScriptの && は「左側がtrueなら右側の値を返す。左側がfalseなら、右側を評価せずfalseのまま返す」という仕組みです。 Reactは真偽値の false・null・undefined を「何も表示しない」として扱うので、結果的に「trueのときだけ右側を表示する」ように見えます。

危険な罠: 左側に「数値の0」を置かない。 {count && <p>件あります</p>} で count が 0 のとき、 && は左側の 0 をそのまま返してしまいます。Reactは数値の 0 を「非表示」とは扱わずそのまま画面に表示するため、 画面にポツンと「0」が出る事故が起きます。対策: count > 0 && のように、必ず真偽値になる条件を左に置きましょう。

04null を return — 何も表示しない

コンポーネントが「ある条件のときは何も表示したくない」場合、null をreturnします。 Reactは null を「ここには何も描かない」という意味として理解します。

function WarningBanner({ warn }) {
  if (!warn) {
    return null; // 何も表示しない
  }
  return <div className="warning">警告です！</div>;
}
実務では稀。 公式ドキュメントでも「コンポーネントから null を返すのは一般的ではない」と説明されています。 たいていは、呼び出す側（親）で「表示するかどうか」を {condition && <Banner />} のように判断する方が見通しが良いことが多いです。 ただし「子の都合で自分自身を消したい」場面では便利な手段として知っておきましょう。

05リストのレンダリング — 配列の map()

「同じ形のカードを、データの数だけ並べたい」——これがリストのレンダリングです。 Day03で学んだ配列メソッド map() が、ここでも主役になります。 「配列の each 要素を、JSX要素に変換する」のが基本パターンです。

const people = ["創造性", "独立思考", "開拓心"];

function List() {
  const listItems = people.map((person) =>
    <li>{person}</li>
  );
  return <ul>{listItems}</ul>;
}
// 結果: <ul><li>創造性</li><li>独立思考</li><li>開拓心</li></ul>

流れを分解すると、こうなります。

  1. 元データ（people という配列）を用意する。
  2. map() で配列の1要素ずつをJSX（<li>）に変換し、新しい配列（listItems）を作る。
  3. その配列を { } でJSXの中に埋め込む。ReactはJSX要素の配列を、自動的に並べて表示してくれます。
JSXの中に直接 {people.map(...)} と書くこともできます。 変数 listItems を経由せず <ul>{people.map((p) => <li>{p}</li>)}</ul> と1行で書く形も非常によく見かけます。意味は同じです。

06key — なぜ必要？ 何を入れる？

上のコードをブラウザの開発者ツールで実際に動かすと、コンソールに Warning: Each child in a list should have a unique "key" prop.（リストの各要素は一意な key を持つべき）という警告が出ます。 リストを作るときは、各要素に key という特別な props を付ける必要があります。

const listItems = people.map((person) =>
  <li key={person.id}>{person.name}</li>
);

key は何のため？

Reactは、リストの中の「どの要素が、前回のどの要素と同じものか」を、画面の見た目ではなく key の値で判断します。 これにより、配列の並び替え・追加・削除が起きたときに、変わっていない要素はそのまま使い、変わった要素だけ更新するという効率的な処理ができます。 （この「並び替えても正しく要素を追跡できる」仕組みについては、state を学ぶ Day08 以降でより実感できます。）

何を key にすればいい？

候補OK?理由
データ自体が持つ一意なID（user.id など）◎ 推奨データベースなどから来る、変わらない一意な値だから
配列のindex（map((p, index) => ...) の index）△ 非推奨並び替え・追加・削除があると同じ要素なのにindexが変わるため、Reactが要素を取り違える
Math.random() など毎回変わる値✕ NG再描画ごとにkeyが変わり、Reactが毎回「全部新しい要素」と誤認する
key はコンポーネントの中から見えない。 key はReact自身が使う特別な情報なので、 props.key として子コンポーネントの中から読み取ることはできません。表示用の値が必要なら、別の名前（例: id）で改めて渡しましょう。

07純粋性（Purity）— コンポーネントは「純粋関数」

ここまでで「条件によって表示を変える」「データから繰り返し表示を作る」を学びました。 最後に、その土台にある大原則を理解しておきましょう。Reactは、コンポーネントを「純粋関数」として書くことを前提に設計されています。

純粋関数（pure function）とは: 数学の関数のように、次の2つの性質を持つ関数です。
  1. 自分のことに集中する。 呼び出される前から存在していた、外側の変数やオブジェクトを変更しない。
  2. 同じ入力には、いつも同じ出力。 同じ Props を渡せば、何度呼ばれても必ず同じJSXを返す。

NGな例: コンポーネントの外の変数を書き換えている

let guest = 0; // コンポーネントの外にある変数

function Cup() {
  guest = guest + 1; // ✕ NG: 外の変数を書き換えている（副作用）
  return <h2>ゲスト# {guest} のお茶</h2>;
}

これだと Cup は呼ばれるたびに違う結果を返してしまい、呼ばれる順番や回数によって表示が変わってしまいます。 これでは「同じ入力には同じ出力」という純粋性が崩れ、予測しづらく不安定なコンポーネントになります。

OKな例: 必要な値を props として受け取る

function Cup({ guest }) {
  return <h2>ゲスト# {guest} のお茶</h2>;
}

function TeaSet() {
  return (
    <>
      <Cup guest={1} />
      <Cup guest={2} />
      <Cup guest={3} />
    </>
  );
}

こちらは guest という props だけを見て、いつも同じ結果を返します。外側の何かを書き換えていないので、何回呼ばれても・どの順番で呼ばれても安心です。

なぜ純粋性が大事？ Reactはパフォーマンスのために、同じコンポーネントを何度も呼んだり、呼ぶ順番を変えたりすることがあります。 コンポーネントが純粋であれば、何度呼んでも・どの順で呼んでも結果は変わらないので、Reactは安心して最適化できます。 逆に不純（外の変数を書き換える等）だと、表示が呼び出し順に依存する不安定なバグを生みます。
「画面の外で何かを起こす」のはダメじゃない。 画面に表示する以外の作用（ネットワーク通信、タイマー、コンソール出力など）を 「副作用（side effect）」と呼びます。これらはコンポーネントの描画そのものの中ではなく、Day14で学ぶ Effect（useEffect） という専用の仕組みの中で行います。 「描画中は副作用を起こさない」が今回のポイントです。
